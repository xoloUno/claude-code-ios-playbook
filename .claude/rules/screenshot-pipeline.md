# Screenshot Pipeline Rule

App Store screenshot pipelines mix automated capture (XCUITest, simctl) with
manual-gesture capture (Live Activity on the lock screen, Home Screen widget,
Control Center). Those two input classes share a directory namespace by default
but want opposite gitignore policies and opposite cadences. This rule is the
canonical reference for how to structure a project's screenshot pipeline so
both flows compose cleanly into ASC-ready submission images.

The pipeline ships as **two named input trees** funneling through a **four-layer
artifact pipeline**, with an **agent-driven loop** that handles the gestures
automation can't do reliably.

## The two-input-tree pattern

A pipeline that mixes automated and manual capture should split inputs into two
named directories with **opposite gitignore policies**:

| Input class | Directory | Gitignored? | Cadence |
|---|---|---|---|
| Manual-gesture captures | `fastlane/manual-captures/<locale>/` | **No** — tracked | Once per release |
| Regenerable app captures | `fastlane/screenshots/<locale>/<device>/raw/` | Yes | Every UI tweak |

The split solves three problems at once:

1. Keeps git history small by ignoring the regenerable bulk.
2. Preserves slow-to-regenerate inputs as a versionable source of truth, so any
   contributor can run the pipeline cold without having you on the keyboard for
   the manual-gesture surfaces.
3. Makes cadence intent legible — "regenerate this every UI tweak" vs.
   "recapture once per release / when the surface changes."

"Once per release" is concretely: when the iOS major version changes, when the
Live Activity / widget / Control Widget UI changes, when a new locale is added,
or when a new highlighted surface is added. Otherwise the manual-captures dir
sits unchanged across builds. Cost benchmark for a 3-locale project: ~19 MB /
9 PNGs at iPhone 17 Pro Max native 1320×2868.

## The four artifact layers

Each `(locale, device, surface)` combination flows through these layers:

```
manual-captures/  ──┐
                    ├──> raw/  ──>  framed/  ──>  composed/
XCUITest captures ──┘
```

### `fastlane/manual-captures/<locale>/`

Manual-gesture inputs captured by the agent-driven loop documented below: the
Live Activity stack on the lock screen, the Home Screen page that hosts the
widget, Control Center pulled down with the Control Widget visible. **Committed
to git.** Recaptured "once per release" — when iOS majors change, when the
Live Activity / widget / Control Widget UI changes, when a new locale is
added, or when a new highlighted surface is added. Never regenerated by an
automated lane: nothing in the project should overwrite this dir except the
agent-driven loop with the user in the loop. Treat these PNGs the way you
treat checked-in design assets — they're inputs, not outputs.

### `fastlane/screenshots/<locale>/<device>/raw/`

XCUITest- or simctl-driven app surfaces (HomeScreen, Picker, etc.). Fast to
regenerate — running the capture lane writes a fresh set in seconds to minutes
depending on screen count. Shifts visually with every UI tweak, which is why
keeping it in git would balloon history. **Gitignored.** This is also where
the Fastfile's `:compose_screenshots` lane stages the tracked
`manual-captures/<locale>/` files using shotsmith's `input_mapping` — so
`90_LockScreen_LiveActivity.png` from manual-captures lands as
`01_LiveActivities.png` (or whatever the canonical caption-order name is) in
`raw/` right before the frame step runs. The two flows merge here.

### `fastlane/screenshots/<locale>/<device>/framed/`

Output of `frames-cli`. Apple device bezels drawn around the raw captures.
**Gitignored.** Regenerable from `raw/` in one shotsmith invocation; never
hand-edit this dir.

### `fastlane/shotsmith/composed/<style>/<locale>/<device>/`

Output of `shotsmith compose`. Gradient backgrounds + captions + final
ASC-ready submission images at the exact 1320×2868 / 2064×2752 dimensions
ASC requires. **Gitignored.** Multiple `<style>` subdirs may exist if the
project keeps several composition presets in flight (mauve, royal-purple,
apple-music, etc.) — only the chosen style gets uploaded.

## Gitignore policy

Add this block verbatim to the project `.gitignore`:

```
# Screenshot pipeline — see .claude/rules/screenshot-pipeline.md
fastlane/screenshots/                # raw + framed (regenerable)
fastlane/shotsmith/composed/         # composed (regenerable)

# Intentionally NOT ignored: fastlane/manual-captures/
# These are tracked manual-gesture inputs (Live Activity stack, Home Screen
# widget, Control Center) — recaptured once per release.
```

The trailing comment block is load-bearing. Make the tracking policy legible
inside `.gitignore` itself, not just in this rule — the next contributor to
audit the gitignore should not have to cross-reference a separate doc to
understand why one fastlane subdir is ignored and an adjacent one isn't.

## The agent-driven capture loop

Manual-gesture surfaces (Live Activity, Home Screen widget, Control Center)
need a person to perform a gesture the simulator can't reliably script. The
canonical contract for that flow is below — this is what the
`/capture-manual-surfaces` slash command implements.

### What the agent does

1. Read the project `config.json` (or equivalent) to determine the locales
   list, target device sim name, bundle ID, and the list of manual surfaces
   to capture.
2. For each `(locale × surface)` tuple, run the per-tuple sequence below.
3. Print a concise summary at the end listing every PNG written.

### Per-tuple sequence

**Step a — Prep (automated, no user gesture).**

```bash
xcrun simctl boot "$DEVICE" 2>/dev/null || true
xcrun simctl bootstatus "$DEVICE" -b >/dev/null
open -a Simulator
```

**Step b — Locale switch (only when changing locales).**

```bash
xcrun simctl spawn "$DEVICE" defaults write -globalDomain AppleLanguages -array "$LOCALE"
xcrun simctl spawn "$DEVICE" defaults write -globalDomain AppleLocale -string "$LOCALE_UNDERSCORED"
xcrun simctl shutdown "$DEVICE"
xcrun simctl boot "$DEVICE"
xcrun simctl bootstatus "$DEVICE" -b >/dev/null
```

This is the **per-sim** language switch. It's distinct from the per-launch-arg
approach used for `raw/` captures (`-AppleLanguages "(es-ES)" -AppleLocale es_ES`
on `simctl launch`). Manual-gesture surfaces are rendered by SpringBoard and
other system processes, not the host app, so the per-launch-arg trick doesn't
reach them — the OS itself has to be in the target locale.

**Step c — Status bar override.** Run the canonical block from
`status-bar-overrides.md`. Every flag matters; do not edit the block here.

**Step d — Launch the host app.**

```bash
xcrun simctl launch "$DEVICE" "$BUNDLE_ID" -WIDGET_DEMO_MODE YES -FASTLANE_SNAPSHOT YES
```

The launch arguments depend on the project; the two above are the common pair
for getting deterministic widget content + suppressing first-run UI. Sleep
briefly (3–5s) afterward so the Live Activity has time to register and the
widget has time to refresh.

**Step e — Pause and prompt the user.**

The agent emits a chat message naming the surface and the gesture, then waits
for the user to type `ready` (or `done`, or any explicit acknowledgment) in
chat. Concretely, the agent does **not** call `read -p` and does **not** try
to script the gesture. It hands off to the user, waits for their reply in the
chat thread, then proceeds.

The wait-for-ready pattern matters because:

- `read -p` interacts badly with `set -e` (EOF on stdin makes the script exit
  silently) and is awkward to wire up cleanly when the script is invoked by
  an agent rather than a TTY.
- Quartz drag automation for Control Center, and `Cmd+L` for Lock Screen, have
  been historically unreliable across simulator versions, OS versions, and
  whether the host terminal has Accessibility permission. The user's hand on
  the keyboard or trackpad is the only durable mechanism.

**Step f — Capture.**

```bash
xcrun simctl io "$DEVICE" screenshot "fastlane/manual-captures/$LOCALE/${NN}_${SURFACE}.png"
```

The filename prefix (`90_…`, `91_…`, `92_…`) is what shotsmith's
`input_mapping` references when staging into `raw/`. Don't renumber casually —
update `input_mapping` in lockstep if you do.

## Surface gesture inventory

These are the canonical manual-gesture surfaces and the gesture the user
performs for each. Add new surfaces with the next available `9X_` prefix.

| Surface | Filename | Gesture user performs |
|---|---|---|
| Live Activity (Lock Screen) | `90_LockScreen_LiveActivity.png` | Lock the device (Cmd+L in Simulator), then verify the Live Activity is rendering on the lock screen. |
| Home Screen widget | `91_HomeScreen_Widget.png` | Unlock (Cmd+Shift+H twice, or swipe up), then navigate to the Home Screen page that contains the widget. |
| Control Center (with Control Widget) | `92_ControlCenter.png` | From any non-app screen, swipe down from the top-right corner of the simulator to invoke Control Center. |

The agent's chat prompt at step (e) should name the surface and read the
gesture verbatim from this table so the user always sees the same instruction.

## Inlined automation script (fallback)

For non-Claude-Code workflows or smoke-testing, here's a standalone bash
script that does prep + status-bar override + launch and waits on stdin
(`echo ready` from another shell, or a key press in interactive use) before
capturing.

`scripts/capture-manual-surface.sh`:

```bash
#!/usr/bin/env bash
# Capture one manual-gesture surface to fastlane/manual-captures/<locale>/.
#
# Pauses for stdin acknowledgment between launch and capture so the human can
# perform the surface gesture (lock screen, home screen widget page, swipe
# Control Center down, etc.).
#
# Usage:
#   scripts/capture-manual-surface.sh <device> <bundle-id> <locale> <surface-name> <output-path>
#
# Example:
#   scripts/capture-manual-surface.sh \
#     "iPhone 17 Pro Max" com.example.app en-US LockScreen_LiveActivity \
#     fastlane/manual-captures/en-US/90_LockScreen_LiveActivity.png

set -euo pipefail

DEVICE="${1:?device name required}"
BUNDLE_ID="${2:?bundle identifier required}"
LOCALE="${3:?locale required (e.g. en-US)}"
SURFACE="${4:?surface name required (e.g. LockScreen_LiveActivity)}"
OUTPUT="${5:?output png path required}"

mkdir -p "$(dirname "$OUTPUT")"

echo "Booting $DEVICE…"
xcrun simctl boot "$DEVICE" 2>/dev/null || true
xcrun simctl bootstatus "$DEVICE" -b >/dev/null
open -a Simulator

# Canonical status-bar override — see .claude/rules/status-bar-overrides.md.
xcrun simctl status_bar "$DEVICE" override \
  --time "9:41" --dataNetwork hide --batteryState charged --batteryLevel 100 \
  --cellularMode active --cellularBars 4 --wifiMode active --wifiBars 3 \
  --operatorName ""

echo "Launching $BUNDLE_ID (locale: $LOCALE)…"
xcrun simctl launch "$DEVICE" "$BUNDLE_ID" \
  -WIDGET_DEMO_MODE YES -FASTLANE_SNAPSHOT YES \
  -AppleLanguages "($LOCALE)" -AppleLocale "${LOCALE//-/_}" >/dev/null
sleep 4  # let widgets / Live Activity register and render

cat <<EOF

================================================================
Surface: $SURFACE   Locale: $LOCALE
Perform the gesture for this surface in the Simulator now.
When the screen shows what should be captured, type 'ready'
and press Enter.
================================================================
EOF

# Wait for explicit acknowledgment. Plain read (no -p) so set -e doesn't
# abort silently on EOF; loop until the user types 'ready' or 'done'.
while IFS= read -r line; do
  case "$line" in
    ready|done|y|yes) break ;;
    *) echo "Type 'ready' (or 'done') and press Enter." ;;
  esac
done

xcrun simctl io "$DEVICE" screenshot "$OUTPUT"
echo "Captured: $OUTPUT"
```

Make executable: `chmod +x scripts/capture-manual-surface.sh`.

## Watch screenshot capture

watchOS screenshots have their own pipeline because Apple's submission rules
and frames-cli's bezel handling differ enough from iPhone/iPad to break the
"raw → framed → composed" assumption. The seven gotchas below are what wall
time gets spent on the first time a project hits the watch pipeline; treat
this as a checklist before shipping.

A future version may inline the watch bezel as a shotsmith subcommand; for
now the watch pipeline ships as separate scripts.

### Checklist

1. **ASC slot dimensions are exact, per device class.** Apple Watch Ultra
   accepts 410×502 OR 422×514 (Apple Watch Ultra 3 simulator captures at
   422×514 native). Other classes have their own dimensions. The full map is
   in `fastlane/deliver/lib/deliver/app_screenshot.rb`'s
   `DEVICE_TYPE_TO_DIMENSIONS`. The composed marketing PNG must be at one of
   those exact sizes — no padding, no scaling-up.
2. **ASC rejects watch screenshots with alpha (PNG32).** Error code
   `IMAGE_ALPHA_NOT_ALLOWED`. iPhone screenshots accept alpha, watch does
   not. ImageMagick's `-alpha off` only marks the channel inactive — to drop
   the alpha channel from the file you need the `PNG24:output.png` prefix in
   the output spec.
3. **frames-cli's watch bezels include band/strap.** Cropping leaves visible
   strap residue or pushes the result outside the 422×514 ASC slot. Don't use
   frames-cli for watch — draw the bezel directly in ImageMagick: concentric
   titanium ring (`#6e6661`) + black inner ring around the rounded watch
   screen.
4. **Sheet auto-presentation needs a launch arg + a deferred Task.** To
   capture a sheet (picker, confirmation), the home view must render BEHIND
   the sheet first. Implement as `-WATCH_SHOW_PICKER` / `-WATCH_SHOW_STOP`
   launch args read in `.onAppear`, then
   `Task { try? await Task.sleep(for: .seconds(0.5)); showPicker = true }`.
   The sleep is mandatory — `.onAppear` fires before SwiftUI has the sheet
   in the view tree.
5. **ScrollViewReader.scrollTo races view layout.** The capture script needs
   ≥5s sleep after `simctl launch -FASTLANE_SNAPSHOT` for demo data to load
   AND the scroll to land. Inside the view, wrap the scroll in
   `Task { try? await Task.sleep(for: .seconds(0.6)); proxy.scrollTo(...) }` —
   same race.
6. **Demo data needs different shapes per screen.** A home view typically
   wants 3 active cards filling the small viewport; a picker wants an empty
   deck so popular items render vibrant/addable, not dimmed-already-added.
   Solve with two functions and route by launch arg in the watch app's
   `.task`.
7. **Locale switching is per-launch-arg, not per-sim.** Pass
   `-AppleLanguages "(es-ES)" -AppleLocale es_ES` to `simctl launch`. No
   reboot needed. (Opposite of the manual-captures flow above, where SpringBoard
   forces per-sim language switch.)

### `scripts/capture-watch-screenshots.sh`

Parameterized by locale and screen. Reads the watch sim name from
`.env.project` as `WATCH_SIM` so the simulator model can be bumped each year
without touching the script.

```bash
#!/usr/bin/env bash
# Capture watchOS screenshots via simctl. Locale + screen parameterized.
#
# Usage:
#   scripts/capture-watch-screenshots.sh <locale> <screen> <output-png>
#
# Reads from .env.project:
#   WATCH_SIM    e.g. "Apple Watch Ultra (49mm)"
#   WATCH_BUNDLE_ID  e.g. com.example.app.watchkitapp
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
[ -f "$ROOT/.env.project" ] && set -a && . "$ROOT/.env.project" && set +a

: "${WATCH_SIM:?WATCH_SIM not set in .env.project}"
: "${WATCH_BUNDLE_ID:?WATCH_BUNDLE_ID not set in .env.project}"

LOCALE="${1:?locale required (e.g. en-US)}"
SCREEN="${2:?screen name required (e.g. home, picker, stop)}"
OUTPUT="${3:?output png path required}"

mkdir -p "$(dirname "$OUTPUT")"

echo "Booting $WATCH_SIM…"
xcrun simctl boot "$WATCH_SIM" 2>/dev/null || true
xcrun simctl bootstatus "$WATCH_SIM" -b >/dev/null   # the trivial fix: wait for full boot
open -a Simulator

# Pick the right launch arg per screen. Two functions in the watch app
# (loadWatchDemoData + loadWatchPickerDemoData) consume these.
case "$SCREEN" in
  home)   EXTRA_ARGS=(-WATCH_DEMO_HOME YES) ;;
  picker) EXTRA_ARGS=(-WATCH_SHOW_PICKER YES) ;;
  stop)   EXTRA_ARGS=(-WATCH_SHOW_STOP YES) ;;
  *) echo "Unknown screen: $SCREEN" >&2; exit 1 ;;
esac

# Locale is per-launch-arg on watch — no per-sim reboot.
echo "Launching $WATCH_BUNDLE_ID (locale: $LOCALE, screen: $SCREEN)…"
xcrun simctl launch "$WATCH_SIM" "$WATCH_BUNDLE_ID" \
  -FASTLANE_SNAPSHOT YES \
  -AppleLanguages "($LOCALE)" -AppleLocale "${LOCALE//-/_}" \
  "${EXTRA_ARGS[@]}" >/dev/null

# 5s — gotcha #5 — demo data load + ScrollViewReader land both fit inside.
sleep 5

xcrun simctl io "$WATCH_SIM" screenshot "$OUTPUT"
echo "Captured: $OUTPUT"
```

### `scripts/compose-watch-marketing.sh`

ImageMagick bezel + gradient + caption composer. Demonstrates the
concentric-rounded-rect titanium ring + black inner ring with an alpha mask
so the watch screen contents render through cleanly.

```bash
#!/usr/bin/env bash
# Compose a watchOS marketing screenshot at an ASC slot dimension.
#
# Usage:
#   scripts/compose-watch-marketing.sh <input-screen-png> <caption> <output-png>
#
# Output is PNG24 (no alpha) at 422x514 — the Apple Watch Ultra 3 ASC slot.
set -euo pipefail

INPUT="${1:?input screenshot png required}"
CAPTION="${2:?caption text required}"
OUTPUT="${3:?output png path required}"

# ASC slot
W=422
H=514

# Watch case rounded-rect geometry inside the canvas
CASE_W=312
CASE_H=380
CASE_X=$(( (W - CASE_W) / 2 ))
CASE_Y=40
CASE_R=68     # outer corner radius
RING_BLACK=8  # black inner ring thickness
SCREEN_PAD=14 # how far the screen sits inside the titanium

TMP="$(mktemp -d)"
trap 'rm -rf "$TMP"' EXIT

# 1. Background gradient (vertical, dusk → coral). Opaque PNG24 from the start.
magick -size ${W}x${H} \
  gradient:'#1B1B2E-#FF6B5C' \
  PNG24:"$TMP/bg.png"

# 2. Titanium outer rounded rect.
magick -size ${CASE_W}x${CASE_H} canvas:none \
  -fill '#6e6661' -draw "roundrectangle 0,0 ${CASE_W},${CASE_H} ${CASE_R},${CASE_R}" \
  PNG32:"$TMP/case.png"

# 3. Black inner ring — same shape, inset by RING_BLACK, draw black on top.
INNER_W=$((CASE_W - 2*RING_BLACK))
INNER_H=$((CASE_H - 2*RING_BLACK))
INNER_R=$((CASE_R - RING_BLACK))
magick "$TMP/case.png" \
  \( -size ${INNER_W}x${INNER_H} canvas:none \
     -fill black -draw "roundrectangle 0,0 ${INNER_W},${INNER_H} ${INNER_R},${INNER_R}" \) \
  -geometry +${RING_BLACK}+${RING_BLACK} -composite \
  PNG32:"$TMP/case_with_ring.png"

# 4. Watch screen — input screenshot, masked to the inner rounded rect.
SCREEN_W=$((INNER_W - 2*SCREEN_PAD))
SCREEN_H=$((INNER_H - 2*SCREEN_PAD))
SCREEN_R=$((INNER_R - SCREEN_PAD))
magick "$INPUT" -resize ${SCREEN_W}x${SCREEN_H}^ -gravity center -extent ${SCREEN_W}x${SCREEN_H} \
  \( -size ${SCREEN_W}x${SCREEN_H} canvas:none \
     -fill white -draw "roundrectangle 0,0 ${SCREEN_W},${SCREEN_H} ${SCREEN_R},${SCREEN_R}" \) \
  -compose DstIn -composite \
  PNG32:"$TMP/screen.png"

# 5. Compose case + screen onto bg.
magick "$TMP/bg.png" \
  "$TMP/case_with_ring.png" -geometry +${CASE_X}+${CASE_Y} -composite \
  "$TMP/screen.png" -geometry +$((CASE_X + RING_BLACK + SCREEN_PAD))+$((CASE_Y + RING_BLACK + SCREEN_PAD)) -composite \
  PNG32:"$TMP/composed.png"

# 6. Caption below the case.
CAPTION_Y=$((CASE_Y + CASE_H + 24))
magick "$TMP/composed.png" \
  -font 'New-York-Small-Bold' -pointsize 26 -fill white \
  -gravity North -annotate +0+${CAPTION_Y} "$CAPTION" \
  PNG32:"$TMP/captioned.png"

# 7. Final flatten to PNG24 — gotcha #2: ASC rejects alpha for watch.
magick "$TMP/captioned.png" -background black -alpha remove -alpha off \
  PNG24:"$OUTPUT"

echo "Composed: $OUTPUT (${W}x${H} PNG24)"
```

Both scripts are starting points — refine the geometry and gradient palette
per project. The structural pieces (boot+bootstatus before launch, per-launch
locale, PNG24 output, concentric rounded-rect bezel) are what's hard to figure
out cold; the cosmetic pieces are easy to iterate on.

## Why this rule exists

A downstream session ran a complete watchOS screenshot pipeline and an
end-to-end recapture of a multi-locale iPhone pipeline back-to-back. Both
sessions hit several of the gotchas above — the unreliable Quartz drag for
Control Center, the per-sim vs per-launch-arg locale split, the ASC alpha
rejection on watch, and the lack of any documented separation between
regenerable and tracked screenshot inputs. The four-layer pattern existed in
practice but lived only in the project's Fastfile. This rule promotes the
durable lessons to the playbook so future projects don't rediscover them.
